## TL;DR

Promptfoo is a CLI and evaluation framework for testing prompts, models, and RAG pipelines before they ship [Getting Started](https://www.promptfoo.dev/docs/getting-started/). The practical pattern is simple: define test cases in YAML, run them against each candidate model, wire the eval into your existing CI pipeline, and fail the build when a prompt change breaks something that used to work [GitHub Action](https://www.promptfoo.dev/docs/integrations/github-action/). This guide walks through that setup end to end, plus what it deliberately doesn't cover.

## Why Does Promptfoo Matter?

If you've shipped an LLM feature, you've probably had this exact experience: you tweak a prompt to fix one issue, ship it, and a week later someone notices it broke a different, unrelated case that used to work fine. Nobody caught it because nobody re-ran the old test cases. There weren't any, or they lived in someone's head.

Promptfoo treats this the way you'd treat any other software regression: write the test once, run it on every change, and fail the build if it breaks. The format is intentionally low-ceremony: YAML configs with prompts, providers, and test cases, with no application code required for basic setups [Getting Started](https://www.promptfoo.dev/docs/getting-started/).

## How Do You Set Up Your First Config?

Install Promptfoo and scaffold a working example:

```bash
npx promptfoo@latest init --example getting-started
cd getting-started
npx promptfoo@latest eval
npx promptfoo@latest view
```

That last command opens a local web viewer showing every prompt/model/test-case combination side by side. Useful for the first pass, before anything is automated.

The core config file is `promptfooconfig.yaml`. Three sections matter:

**Prompts**: your templates, with `{{variable}}` placeholders:

```yaml
prompts:
  - 'Convert the following English text to {{language}}: {{input}}'
```

**Providers**: every model you want tested, run side by side:

```yaml
providers:
  - openai:chat:gpt-5.4
  - openai:chat:gpt-5.4-mini
  - anthropic:messages:claude-opus-4-6
  - google:gemini-3.1-pro-preview
```

This is the detail that matters most for enterprise decision-making: you're not betting on one vendor's benchmark claims. You run your actual prompts, on your actual test cases, across every model you're considering, and get a comparison table back [Getting Started](https://www.promptfoo.dev/docs/getting-started/).

**Tests and assertions**: the actual pass/fail logic:

```yaml
tests:
  - vars:
      language: French
      input: Hello world
    assert:
      - type: contains
        value: 'Bonjour le monde'
  - vars:
      language: Spanish
      input: Where is the library?
    assert:
      - type: icontains
        value: 'Dónde está la biblioteca'
```

## Which Assertions Should You Use?

This is where most teams under-invest. The assertion types map onto real production failure modes:

- `contains` / `icontains` - did the output include something it must include [Assertions](https://www.promptfoo.dev/docs/configuration/expected-outputs/)
- `llm-rubric` - use a model to grade free-form instructions you can't check with a regex, e.g. "do not mention you are an AI assistant" or "explain the answer without apologizing" [Assertions](https://www.promptfoo.dev/docs/configuration/expected-outputs/)
- `cost` - fail if a call costs more than a threshold; useful when a "better" model also happens to be more expensive per call [Assertions](https://www.promptfoo.dev/docs/configuration/expected-outputs/)
- `latency` - fail if a response takes longer than your SLA allows [Assertions](https://www.promptfoo.dev/docs/configuration/expected-outputs/)

A real example combining both cost and latency gates:

```yaml
defaultTest:
  assert:
    - type: cost
      threshold: 0.002
    - type: latency
      threshold: 3000
```

For RAG pipelines specifically, Promptfoo supports factuality, answer relevance, context recall, context relevance, and context faithfulness checks [Assertions](https://www.promptfoo.dev/docs/configuration/expected-outputs/). Those are the things that actually go wrong in retrieval systems, rather than a generic "does this look right" check.

For agent workflows, the equivalent is testing tool use and response quality across multi-turn scenarios, checking whether the agent actually completed the task and followed the trajectory you expected, not just whether the final answer looked plausible.

## Step 3: Wire It Into CI

This is the step that turns a useful local tool into an enterprise safety net. The official GitHub Action automatically runs a before/after comparison whenever a prompt file changes in a pull request, then posts the results as a PR comment:

```yaml
name: 'Prompt Evaluation'

on:
  pull_request:
    paths:
      - 'prompts/**'

jobs:
  evaluate:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - name: Set up promptfoo cache
        uses: actions/cache@v4
        with:
          path: ~/.cache/promptfoo
          key: ${{ runner.os }}-promptfoo-v1
          restore-keys: |
            ${{ runner.os }}-promptfoo-

      - name: Run promptfoo evaluation
        uses: promptfoo/promptfoo-action@v1
        with:
          openai-api-key: ${{ secrets.OPENAI_API_KEY }}
          github-token: ${{ secrets.GITHUB_TOKEN }}
          prompts: 'prompts/**/*.json'
          config: 'prompts/promptfooconfig.yaml'
          cache-path: ~/.cache/promptfoo
```

Three practical notes if you're setting this up for real:

1. Cache it. The action caches LLM requests and outputs between runs, which saves real money on repeated test cases [GitHub Action](https://www.promptfoo.dev/docs/integrations/github-action/).
2. Scope the trigger narrowly. The `paths: prompts/**` filter means this only runs when prompt files actually change, not on every unrelated commit [GitHub Action](https://www.promptfoo.dev/docs/integrations/github-action/).
3. If you use another CI system, mirror the same pattern there. Promptfoo documents the GitHub workflow as the reference setup, but the underlying idea is still the same: run evals before merge [GitHub Action](https://www.promptfoo.dev/docs/integrations/github-action/).

## What Should Block a Merge?

Not every assertion should be a hard gate. A practical split enterprise teams land on:

- Blocking: correctness assertions (`contains`, `llm-rubric` checks against known-bad outputs), cost thresholds that would blow a budget, latency thresholds that would breach an SLA
- Reporting only: comparative model-quality scores you're tracking over time but haven't decided to act on yet

The GitHub Action's PR-comment behavior already supports this split naturally. The comment shows the full comparison, but whether the build actually fails is a property of your assertions, not the integration itself [GitHub Action](https://www.promptfoo.dev/docs/integrations/github-action/).

## What Won't This Setup Catch?

Worth being direct about the boundary here, since it shapes what else you'll need:

- It tests pre-deployment, not production. Promptfoo evaluates prompts before they ship. It doesn't score live traffic, detect quality drift after deployment, or alert you when real user inputs start hitting edge cases your test suite didn't anticipate. If you need that, you're looking at a separate production-observability layer on top of this, not instead of it.
- YAML scales adequately, not gracefully. A test suite with a few dozen cases is easy to review. A few hundred cases across multiple prompts and models starts to get unwieldy to review in raw YAML. Worth a basic naming/folder convention from day one rather than retrofitting one later.
- It's CLI-first. Reviewing eval results and contributing new test cases is comfortable for engineers and awkward for product managers, QA leads, or domain experts who'd want to weigh in on what "correct" looks like but don't want to touch a terminal. The web viewer (`promptfoo view`) helps, but it's still adjacent to a command-line workflow, not a replacement for one.

None of that makes the tool wrong for this job. Pre-deployment regression testing is exactly the job it's built for. It just means "set up Promptfoo in CI" answers one real question and deliberately leaves a different one for something else.

## Where Should You Start First?

Don't try to write the comprehensive test suite on day one. The lowest-friction entry point:

1. Pick one prompt that's already caused a production incident or near-miss.
2. Write 5-10 test cases covering the failure you had plus a few cases that used to work correctly.
3. Run it locally against your current model and confirm it passes.
4. Add the GitHub Action, or your CI equivalent, scoped to just that prompt's file path.
5. Expand coverage prompt by prompt, as each one earns its place from an actual incident or a deliberate model-comparison decision, not preemptively for everything at once.

That gets you a real regression net around the thing that already bit you, without trying to formalize testing for prompts nobody has had a problem with yet.

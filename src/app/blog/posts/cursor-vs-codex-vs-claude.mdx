## What This Is Not

I work with three AI coding tools in production — Cursor, OpenAI Codex, and Claude Code. That is not an argument for buying all three. It is an account of how the *nature of the work* picks the tool, and why most weeks I never leave Cursor.

This is not a benchmark post. I am not scoring autocomplete quality, coding dataset results, or which assistant writes the cleanest Python. Copilot, Windsurf, Devin, and the rest deserve their own write-ups.

If you build agent systems, integrate LLM SDKs, and ship to production, the useful question is not which model wins a leaderboard. It is which workflow removes your current bottleneck — implementation, delegation, or understanding.

## Skip the Feature Matrix

Feature checklists go stale fast. These products converge constantly. What lasts is comparing **workflows**, not product labels.

Three questions I return to:

- **Where do I wait?** Cursor is optimized for keeping you in the implementation loop, even though it now supports background Cloud Agents. A *cloud task* — an agent running remotely against your repo and returning a reviewable pull request — means you context-switch and come back later.
- **What's the review burden?** How much code you need to read before you are comfortable merging.
- **Where does context break?** Mid-session when the agent forgets an earlier decision, or across an async handoff when the PR does not capture your intent.

| Best fit | Cursor | Codex | Claude Code |
|---|---|---|---|
| **Primary workflow** | Implementation | Delegation | Understanding |
| **Best when** | You already know what to build | The work is repetitive and well specified | You don't yet understand the system |
| **Review effort** | Low | High | Medium |

## Cursor: Your Default Environment

Most of my week lives in Cursor — APIs, SDK integrations, production debugging, tests, review of AI-generated diffs. It stays open because the edit-review loop is tight: I spend more time solving problems than steering the assistant.

After an SDK rollout, our eval pipeline started failing while unit tests stayed green. The contract had changed — structured output no longer matched our Pydantic model. I traced call sites, fixed the schema, reran evals, without leaving the editor.

That's where Cursor adds value. Once I understand the problem, I move from idea to validation without breaking flow.

The weakness is the mirror image: easy to stay too long. Some work belongs in delegation or understanding-first. Knowing when to close Cursor matters as much as knowing how to use it.

AI tools implement architecture well; they don't invent it. Wrong boundaries or retry policy produce code that faithfully encodes a bad design.

## Codex: Bandwidth, Not Judgment

I open Codex after decisions are made — when what remains is repetitive implementation that's easy to review. Cursor helps me think while I code. Codex helps me stop thinking about work that doesn't need my attention.

SDK migration: pattern documented, reference implementation shipped, dozens of files left. I wrote the spec, started a remote task, went back to Cursor. The PR was largely correct; review caught an old import and a missed async context manager. Still beat making the same edit dozens of times.

Codex fits migrations, test updates, import sweeps, lint — anything with a clear definition of done.

If you can't describe the task precisely enough for another engineer, don't hand it to an agent.

## Claude Code: The System, Not the Syntax

Claude Code fits when comprehension is the bottleneck, not typing speed. On inherited codebases I use it to map responsibilities, trace how requests move through services, and locate where state and retries actually live — before I commit to a change.

It is a poor fit for one-line edits or fast implementation loops. It does not replace architectural judgment. When I need repository archaeology, though, it usually gets me to "I know where to cut" faster than searching blindly in an editor. Once I know where the change belongs, I switch back to Cursor. Understanding is the hard part; implementation is usually straightforward.

## What Changed in My Workflow

The biggest surprise was not that AI writes code. It was how my time got reallocated.

Two years ago I mostly wrote code. Today I define interfaces, write evaluation criteria, review generated diffs, and debug production behavior. I type less. The judgment load did not shrink.

The hidden cost is not the subscription. It is **review**.

Every delegated change is someone else's code. That review time does not vanish — it often grows, because you were not in the room while the diff was generated. I grep async/sync mismatches first, run evals before merging prompt or tool changes, and treat clean PRs as drafts until traces and tests say otherwise.

Eval scores and traces became the gate. A tidy diff stopped being enough.

## How to Choose

I no longer choose tools based on the language or framework. I choose them based on what is slowing me down.

| If my bottleneck is... | I reach for... |
|---|---|
| I already understand the problem and need to implement it quickly | **Cursor** |
| The work is repetitive, well-scoped, and easy to review | **Codex** |
| I need to understand an unfamiliar system before making changes | **Claude Code**, then **Cursor** for implementation |

The pattern is simple: **Implementation → Cursor · Delegation → Codex · Understanding → Claude Code**

Language and framework do not drive the switch. The **nature of the work** does.

Most weeks I stay in Cursor. Codex and Claude Code appear only when the problem genuinely needs delegation or deeper understanding.

## What Actually Differentiates These Tools

AI coding tools are converging. Every few months they borrow another feature from each other. What continues to matter isn't the feature list — it's the **workflow each tool is designed to optimize**, and what that does to **you**.

Engineering is shifting from primary author to spec writer, reviewer, and system owner. Agents edit files faster than any team could by hand. They cannot tell you which failure modes matter, whether an eval regression is acceptable, or when to stop adding tool calls.

The best AI engineers I know do not collect the most subscriptions. They know which problems need implementation, which need delegation, and which deserve an hour of understanding before a line changes.

The question I ask myself is no longer "Which AI tool should I use?" It's "What is the bottleneck in front of me?"

That is a workflow decision — not a model decision.

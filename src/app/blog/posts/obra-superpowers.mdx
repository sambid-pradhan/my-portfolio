AI coding agents rarely fail because they're incapable—they fail because they start coding before they understand the problem. [Obra Superpowers](https://github.com/obra/superpowers) fixes that by enforcing an engineering workflow: design before code, plans before implementation, TDD before production, and systematic debugging before fixes.

**TL;DR:** Install it for your coding agent (Cursor, Claude Code, Codex CLI, Gemini CLI, and others). Skills activate automatically—brainstorming before features, written plans before code, strict TDD during implementation, systematic debugging when things break. It feels slower at first; the code is smaller, more testable, and less likely to need a rewrite.

*Assumes you use a coding agent with skill or plugin support. June 2026—check [obra-superpowers.mintlify.app](https://obra-superpowers.mintlify.app) for per-platform install steps.*

## The Problem

Say "add user authentication." The agent ships 400 lines of plausible code—wrong session strategy, ignored patterns, zero tests. The model isn't the bottleneck. The process is.

Built by [Jesse Vincent (obra)](https://github.com/obra), Superpowers loads **skills**: markdown files with hard gates and enforced workflows. The TDD skill deletes code written before tests. The brainstorming skill blocks implementation until you approve a design. Default prompts inform; Superpowers skills enforce.

Superpowers does not replace your editor or agent. It adds a process layer—the difference is whether the agent improvises or follows a defined engineering workflow.

## What It Is

Superpowers bundles three things: a **skills library** (14+ composable `SKILL.md` files), **workflow orchestration** (skills activate in sequence by task type), and **platform adapters** (same skills in Cursor, Claude Code, Codex CLI, Gemini CLI, and others).

The core insight: **a skill is a process with enforcement**, not a prompt snippet. Skills are plain markdown with checklists, hard gates, and stop conditions—they load automatically when the agent detects feature work, bugfixes, or plan execution. Activation is not perfect; invoke a skill by name if the wrong one loads.

## Installation

Start with the [platform install guides](https://obra-superpowers.mintlify.app). **Cursor** — use agent chat, not inline completions:

```text
/plugin-add superpowers
```

Restart the session, then verify with `Do you have superpowers?` The agent should list available skills.

## The Workflow

Superpowers replaces "ask and pray" with a pipeline. Each phase has a skill, a deliverable, and a gate you control.

![The Superpowers Workflow — six-phase pipeline from brainstorming through merge with confidence](/images/blog/obra-superpowers-timeline.png)

1. **Brainstorming** — design spec to `docs/superpowers/specs/`; you approve before code starts
2. **Writing plans** — 2–5 minute tasks with file paths, tests, and verification steps
3. **TDD** — no production code without a failing test first (RED → GREEN → REFACTOR)
4. **Subagent execution** — fresh subagent per task, two-stage review (spec compliance, then code quality)
5. **Debug & verification** — root cause before fixes; real commands before calling it done

Optional: isolated Git worktree for feature development away from main.

## Teams, Not Just Individuals

![Team workflow without vs with Superpowers — shared skills lead to predictable code reviews](/images/blog/obra-superpowers-team-flow.svg)

Without Superpowers, each engineer prompts differently—big-bang implementations, skipped tests under pressure, code review as process negotiation. With it, everyone runs the same skills. Specs and plans live in the repo. Onboarding means pointing at the skills library, not re-explaining your prompting style.

## Example: Building Authentication

Same prompt, same repo, same afternoon — with and without Superpowers on a Next.js app:

| | Without Superpowers | With Superpowers |
|---|---|---|
| **Lines changed** | ~450 | ~110 |
| **Tests** | 0 | 18 |
| **Session strategy** | Wrong JWT, ignored existing cookie pattern | Locked in design spec before files changed |
| **Time to mergeable PR** | ~2 hours of rework | First PR merged after review |

Brainstorming surfaced the cookie-vs-JWT question before any files changed. TDD produced 18 tests. Review was diff-sized, not archaeology.

## Who Should Use Superpowers?

For work where **process quality determines outcome quality**—not one-line fixes or inline completions.

| You | Why it fits |
|---|---|
| **Solo dev shipping features** | Catches architecture mistakes before files change |
| **Tech lead / architect** | Specs in repo = shared standard, not personal prompting |
| **Team on agent chat daily** | Same skills across platforms, predictable reviews |
| **Dev burned by agent rework** | Gates exist if you rewrote plausible code last session |

**Skip it when:** throwaway spike (say so explicitly), no test infrastructure yet, or a typo fix. Superpowers is composable—you do not need every phase on every task.

**Start with design intent** — "Let's design a contact form with server-side validation," not "Add a contact form."

**Approve explicitly** — say "approved" or push back before code starts.

**Commit specs and plans** — they become context for future sessions.

## Limitations

- Skills enforce process; you still decide architecture and what ships
- TDD needs an existing test framework—set up infrastructure first
- First session feels heavy; payoff shows up around session three
- Automatic activation is imperfect—restart or invoke by name

## Conclusion

The model was never the bottleneck. The process was. Superpowers makes the session **accountable**: a spec you approved, tests before production code, verification before "done."

That's why one authentication feature ended at ~110 lines with 18 tests instead of ~450 lines of rework. The workflow didn't make the model smarter—it made the engineering process harder to skip.

The agents are already capable enough. What most of us lack is the discipline to make them prove it.

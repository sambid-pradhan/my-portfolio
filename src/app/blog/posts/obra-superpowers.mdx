**TL;DR:** [Obra Superpowers](https://github.com/obra/superpowers) gives coding agents a real software engineering workflow instead of ad-hoc improvisation. Install it in Cursor with `/plugin-add superpowers`, then let skills activate automatically: brainstorming before features, written plans before code, strict TDD during implementation, and systematic debugging when things break. It feels slower at first. The code is smaller, more testable, and less likely to need a full rewrite.

*This post assumes you already use Cursor Agent or a similar coding agent. Written June 2026 — install steps and skill names may change; check [obra-superpowers.mintlify.app](https://obra-superpowers.mintlify.app) for current docs.*

## The Problem Superpowers Solves

Most coding agents share the same failure mode: they start typing before they understand the problem.

You say "add user authentication." The agent creates files, wires middleware, adds dependencies, and produces 400 lines of plausible-looking code. Then you discover it picked the wrong session strategy, ignored your existing auth patterns, and wrote zero tests.

The model is not the bottleneck. The process is.

Superpowers — built by [Jesse Vincent (obra)](https://github.com/obra) — treats agent behavior as an engineering problem. Instead of hoping the model "knows" good practices, it loads **skills**: markdown instruction files with hard gates, checklists, and enforced workflows. Skills are not suggestions. The TDD skill says delete code written before tests. The brainstorming skill blocks implementation until you approve a design.

Think of it as the difference between a speed limit sign and a speed bump. Default agent prompts inform. Superpowers skills enforce.

## What Superpowers Actually Is

Superpowers is three things bundled together:

| Component | What it does |
|---|---|
| **Skills library** | 14+ composable `SKILL.md` files (brainstorming, TDD, debugging, planning, code review) |
| **Workflow orchestration** | Skills activate in sequence based on what you're doing |
| **Platform adapters** | Same skills work in Cursor, Claude Code, Codex CLI, Gemini CLI, and others |

Skills are plain markdown. Any agent that can read a skill file can follow it. The same skills work across Cursor, Claude Code, and Codex CLI without retraining your workflow.

The core insight: **a skill is a process with enforcement**, not a prompt snippet. The TDD skill does not suggest writing tests first. It mandates red-green-refactor and instructs the agent to delete any code written before a failing test exists.

### Why skills activate — and why that works

Skills activate automatically when the agent detects a matching task type. That sounds like magic until you understand the mechanism.

Each skill is a structured markdown file that encodes a workflow — checklists, hard gates, and explicit stop conditions. When the agent detects a matching task (feature work, bugfix, plan execution), it loads the skill into its working context. That changes behavior from free-form generation to following a defined engineering process. The brainstorming skill injects "ask before you code." The TDD skill injects "delete code written before tests." The debugging skill injects "find root cause before proposing fixes."

Automatic activation is not perfect — you can always invoke a skill explicitly — but it removes the burden of remembering which process applies to which task.

## Installation

Use **Agent chat** (`Ctrl+L` / `Cmd+L`), not Composer or inline completions.

```text
/plugin-add superpowers
```

Restart Agent chat, then verify:

```text
Do you have superpowers?
```

Done. The agent should confirm and list available skills. To update: `/plugin-update superpowers`. Troubleshooting: [Cursor install guide](https://obra-superpowers.mintlify.app/installation/cursor).

## The Superpowers Workflow

Superpowers replaces "ask and pray" with a pipeline. Each phase has a skill, a deliverable, and a gate you control.

![Superpowers session workflow — User through Brainstorm, Approve, Plan, Task, Test, Implementation, Review, Deploy](/images/blog/obra-superpowers-session-flow.svg)

*Edit source: [`obra-superpowers-session-flow.drawio`](/diagrams/obra-superpowers-session-flow.drawio)*

**Brainstorming** runs before any feature work. The agent explores your project, asks clarifying questions one at a time, proposes 2–3 approaches, and writes a spec to `docs/superpowers/specs/`. No implementation until you approve — not even for "simple" tasks.

**Writing plans** breaks approved specs into bite-sized tasks (2–5 minutes each), each specifying files, tests, and expected output. Plans save to `docs/superpowers/plans/` with enough detail that a subagent can execute without improvising.

**TDD** enforces one iron law: no production code without a failing test first. RED → verify failure → GREEN → verify pass → REFACTOR → commit. Wrote code before the test? Delete it. Start over.

For larger work, **subagent-driven-development** runs the plan task by task with two-stage review. On a 12-task portfolio feature, all tasks completed in one session without drifting from the approved spec.

When things break, **systematic-debugging** mandates root-cause investigation before fixes. **verification-before-completion** runs real commands before claiming done.

## Teams, Not Just Individuals

For architects and tech leads, the value is not only personal productivity — it is **team consistency**.

![Team workflow without vs with Superpowers — shared skills lead to predictable code reviews](/images/blog/obra-superpowers-team-flow.svg)

*Edit source: [`obra-superpowers-team-flow.drawio`](/diagrams/obra-superpowers-team-flow.drawio)*

Without Superpowers, each engineer prompts differently. Developer A favors big-bang implementations. Developer B skips tests under deadline pressure. Code review becomes a negotiation about process, not just code quality.

With Superpowers, every engineer runs the same skills — same brainstorming gate, same plan format, same TDD cycle, same verification step. Specs and plans live in the repo. Reviewers know what to expect. Onboarding a new team member means pointing them at the skills library, not re-explaining your personal prompting style.

That positions Superpowers as a **team engineering standard**, not a personal hack.

## How to Use It Effectively

Start with design intent, not implementation. "Add a contact form" produces immediate file edits. "Let's design a contact form with server-side validation and no third-party services" triggers brainstorming.

Approve designs explicitly — say "approved" or push back. Commit specs and plans to the repo; they become context for future sessions.

When stuck, process skills win: "Let's build X" → brainstorm first. "Fix this bug" → debug first. User instructions always override — say "this is a throwaway spike" if you genuinely need to skip the workflow. Use Agent chat (`Ctrl+L`), not Composer.

## Example: Building Authentication

Same prompt, same repo, same afternoon — with and without Superpowers. Numbers from my own session on a Next.js app with existing middleware patterns.

| | Without Superpowers | With Superpowers |
|---|---|---|
| **Lines changed** | ~450 | ~110 |
| **Tests** | 0 | 18 |
| **JWT strategy** | Wrong — stateless JWT, ignored existing session cookie | Matched existing pattern, locked in design spec |
| **Time to mergeable PR** | ~2 hours of rework | First PR merged after review |

Without Superpowers, the agent started coding immediately. Plausible output, wrong architecture, no tests — rework dominated the session.

With Superpowers, brainstorming surfaced the session strategy question before any files changed. The spec locked cookie vs JWT, middleware scope, and test coverage. TDD produced 18 tests across auth middleware, token refresh, and protected routes. Review was diff-sized, not archaeology.

That gap — 450 lines of fix-it vs 110 lines of ship-it — is what "less rework" actually means.

## Current Limitations

Superpowers is not a silver bullet. Be honest about what it does not do:

- **Does not replace engineering judgment.** Skills enforce process; you still decide architecture, trade-offs, and what ships.
- **TDD requires an existing test framework.** If the project has no tests, set up infrastructure first — or the TDD skill will fight you.
- **Can feel heavy for rapid prototypes.** A throwaway spike does not need a full design spec. Say so explicitly.
- **Learning curve for first-time users.** The first session feels like more process. The payoff shows up on session three.
- **Automatic activation is not perfect.** If the wrong skill loads — or none loads — restart the session or invoke the skill by name.

One-line typo fixes and inline completions do not need Superpowers. The ceremony pays off on features that matter.

## Conclusion

Obra Superpowers does not make your agent smarter. It makes your agent **more disciplined**. Install it with `/plugin-add superpowers`, start feature work with design intent, approve specs before code, and let TDD run at least once on something that matters.

The first session feels like more process. The third session produces code you actually want to merge.
